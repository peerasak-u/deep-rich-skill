# Deep Rich Report Design System

Use this design reference whenever a workflow creates an HTML report, dashboard, company profile, review artifact, or printable decision brief.

## Purpose

Deep Rich reports should feel like one product, even when generated by different agents. The visual language is:

- calm private-banking dashboard, not trading terminal
- light theme by default, matching `dashboard/index.html`
- rounded cards, soft borders, pill navigation, and clear section hierarchy
- facts first; estimates and interpretation must be visually labeled
- portfolio values in THB, with native currency shown for US and crypto positions

## File locations

Generated app artifacts belong in the Deep Rich app root, not this skill package:

| Artifact | Path pattern | Notes |
|---|---|---|
| Portfolio dashboard | `dashboard/index.html` or `dashboard/dashboard.html` | Main portfolio surface |
| Company profile HTML | `company/<symbol-lower>.html` | User-facing company report |
| Company profile data | `.deep-rich/companies/<SYM>.json` | Reusable structured data |
| Review artifacts | `.deep-rich/reviews/` | Private review output |

Do not create private portfolio artifacts inside this skill repo.

## Required visual components

Every major HTML report should include:

1. **Top bar** — product name, artifact type, update time, data source.
2. **Pill navigation** — anchor links to major sections.
3. **Hero decision card** — the main conclusion in one sentence.
4. **Fact strip** — 3 to 5 key metrics with dates/currencies.
5. **Two-column focus layout** — main narrative plus numeric/visual support.
6. **Tables for comparable data** — competitors, holdings, financials, risks.
7. **Callouts** — at least one guardrail or caveat.
8. **Data-quality boundary** — facts vs calculations vs estimates vs unknowns.
9. **Company logos** — use downloaded local `.deep-rich` logo assets for stock/company rows and hero cards when available.

## Logo usage in HTML workflows

The review, dashboard, and company profile HTML workflows must use locally downloaded logos from the Deep Rich app root. Do not hotlink logo provider URLs directly in generated reports.

| Workflow | Logo usage |
|---|---|
| Review HTML | Show local logos in profit-taking/loss-cutting cards, holdings action cards, and holdings tables. |
| Company profile HTML | Show local logo in the hero card; fallback to a generated ticker tile only if no logo is cached. |
| Dashboard HTML | Show local logos in holdings tables/cards and any top-position widgets. |

Expected cache path:

```text
.deep-rich/company-assets/logos/<symbol-lower>.<ext>
```

Expected profile metadata:

```json
{
  "logo": {
    "path": ".deep-rich/company-assets/logos/net.png",
    "source": "official_website | benzinga | logo_dev | companylogo_xyz | logokit | generated",
    "source_url": "https://...",
    "fetched_at": "YYYY-MM-DD",
    "license_note": "internal report use; verify before redistribution",
    "confidence": "high | medium | low"
  }
}
```

HTML exporters should resolve logo paths relative to the generated HTML file. For example, from `company/net.html`, `.deep-rich/company-assets/logos/net.png` is typically referenced as `../.deep-rich/company-assets/logos/net.png`. If the artifact is fully self-contained, embed the cached local image as a small data URI.

Fallback order for renderers:

1. Use `.deep-rich/companies/<SYM>.json.logo.path` when present and file exists.
2. Try `.deep-rich/company-assets/logos/<symbol-lower>.png`, `.webp`, `.jpg`, `.svg`.
3. Use generated ticker/initials tile.

Missing logos must not block report generation, but the data-quality section should list `logo` as unknown/missing when no trusted source was cached.

## Company profile template workflow

Company profile HTML should be generated from [templates/company-profile.html](templates/company-profile.html), not hand-written per company.

Division of responsibility:

| Actor | Responsibility |
|---|---|
| Agent / LLM | Research facts, fill `.deep-rich/companies/<SYM>.json`, label facts vs estimates, choose thesis/verdict |
| Script / exporter | Load the template, escape values, render lists/tables, resolve local logo paths, inject Chart.js data, write `company/<symbol>.html` |
| Template | Own the visual style, layout, sections, Chart.js CDN usage, and fallback structure |

This keeps style consistent while still letting the agent do the research and judgment work.

## Company profile visual pattern

A company profile should follow this order:

1. Header: company name, symbol, exchange, sector, update time.
2. Position snapshot when held: quantity, cost basis, current value, gain/loss, THB conversion.
3. Company overview: business description, products, revenue model, moat.
4. Company logo: use a trusted local cached logo when available; otherwise use a consistent ticker/initials tile. See [DATA-SOURCES](DATA-SOURCES.md#company-logos).
5. Evidence map: use readable evidence bars/cards by default. Avoid radar/polar label layouts in generated static HTML because labels can overlap at different viewport widths. If using a snowflake-inspired shape, provide enough space and verify screenshots at desktop and mobile widths. Always label it as Deep Rich evidence, not Simply Wall St scoring.
5. Valuation guardrail: factual multiples only unless a transparent model exists.
6. Future growth: clearly labeled estimate section.
7. Past performance: historical revenue, margin, cash flow, price range.
8. Financial health: balance sheet, debt/equity, cash/runway where available.
9. Thesis: bull case, bear case, verdict, suggested action.
10. Risk analysis: risk flags, catalysts, open monitoring questions.
11. Management and ownership: include when available; mark unknown otherwise.
12. Data status: source, last updated, stale-data warning, missing fields.

## Chart libraries

CDN chart libraries are allowed for generated HTML reports because these artifacts are optimized for easy reading by the user and agents normally have internet access.

Preferred options:

| Library | Use for |
|---|---|
| Chart.js CDN | Radar/evidence maps, allocation doughnuts, drift bars, small trend charts |
| Mermaid CDN | Technical diagrams only when a flow/sequence/relationship diagram is clearer |

Rules:

- Use CDN only for visualization code, never for private data upload.
- Keep all private portfolio/company data embedded locally in the HTML or loaded from local `.deep-rich` artifacts.
- Add a readable HTML fallback near important charts when the chart carries decision-critical information.
- Keep charts restrained: they should clarify the report, not turn it into a trading terminal.
- For Chart.js, prefer this CDN by default: `https://cdn.jsdelivr.net/npm/chart.js`.

## Visual rules

- Use light dashboard theme by default.
- Use dark only when explicitly making a presentation/report outside the app dashboard context.
- Keep dense analysis in cards and tables, not long paragraphs.
- Use green for strength, yellow for caution, red/coral for risk, cyan for neutral information.
- For chip/tag rows, set the flex container to `align-items: flex-start`; otherwise a long wrapped chip can stretch every sibling chip into tall pills.
- Keep short chips on one line with `white-space: nowrap`; give long chips an explicit `.long` class or max-width so wrapping is intentional.
- Do not imply precision that data does not support.
- Do not copy Simply Wall St branding, exact UI, or proprietary scoring. Use it only as information-architecture inspiration.

## Copy rules

Use these labels consistently:

| Label | Meaning |
|---|---|
| Fact | Directly sourced and dated |
| Calculated | Derived from visible inputs or app scripts |
| Estimate | Analyst, management guidance, TAM, or forecast |
| Interpretation | Agent synthesis from facts |
| Unknown | Missing, stale, or not yet researched |

Never present forecasts, fair value, analyst target prices, TAM, or future margins as facts.
